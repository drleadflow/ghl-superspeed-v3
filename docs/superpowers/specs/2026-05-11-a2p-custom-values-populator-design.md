# A2P Custom-Value Populator — Design

**Date:** 2026-05-11
**Status:** Approved (design); pending implementation plan

## Problem

Onboarding a client for A2P (10DLC) SMS registration requires a public-facing
website with a home page, privacy policy, and terms of service that reflect the
client's business. We already maintain a fixed 3-page GHL funnel ("A2P Landing
Page (UPDATED 2026)") that is cloned onto each client sub-account from an agency
snapshot. **The pages never change** — every dynamic piece is a
`{{ custom_values.* }}` merge tag. So onboarding a sub-account for A2P reduces
to: set ~18 custom values correctly on that location.

Today that is done by hand. This skill automates it: given a business URL, a
sub-account ID, and (optionally) a vault brief, it scrapes/derives the facts,
generates the on-brand copy, and writes the 18 custom values via the GHL API.

## Scope

**In scope:** populate exactly the 18 custom values listed below on a given GHL
location, creating any that are missing, updating the rest, never overwriting a
non-empty value with a blank.

**Out of scope:** the funnel pages, the opt-in form and its checkbox labels, the
other ~58 custom values on the location, GHL workflows, and the actual Twilio
A2P brand/campaign submission. (The opt-in form's consent text uses
`{{ custom_values.business_profile__legal_business_name }}` and
`{{ custom_values.business_short_name }}`, so populating those two values is
expected to flow through to the form automatically; verifying that GHL renders
custom-value tags in form checkbox labels is a research item in the plan, not a
deliverable here.)

## The 18 custom values

Matched on the location by **`fieldKey`** (the `{{ custom_values.<slug> }}`
slug), not by id (ids are per-location). When *creating* a missing value, use
the canonical display name from this table; when *updating* an existing one,
keep whatever name is already on the location.

### Facts — resolved deterministically, priority: brief → location API → site

| slug | canonical name | source |
|---|---|---|
| `business_short_name` | Business Short Name | brief; else site `<title>` / brand name |
| `business_profile__legal_business_name` | Business Profile - Legal Business Name | brief; else **leave blank + flag** ("set manually from the Twilio registration") |
| `business_profile__phone_number` | BUSINESS PROFILE - Phone Number | brief; else location's purchased phone number via API; else flag |
| `from_email` | From Email | brief; else `info@<domain>`; else flag |
| `logo` | Logo | literal string `{{location.logo_url}}` (passthrough; matches the reference location) |
| `primary_color` | Primary Color | best-effort hex from site CSS; blank if not confident |
| `secondary_color__hex_code` | Secondary Color - Hex Code | best-effort hex from site CSS; blank if not confident |

### Copy — generated by Claude inside the skill, from scraped site + brief, in DLF brand voice

| slug | canonical name | notes |
|---|---|---|
| `about_us__description` | About Us - Description | short paragraph |
| `business_welcome_message` | Business Welcome Message | 1–2 sentences, SEO-flavored intro |
| `business_solution__description` | Business Solution - Description | tagline-style line |
| `our_services_1` | Our Services 1 | service name / headline line |
| `our_services_1__description` | Our Services 1 - Description | one-sentence blurb |
| `our_services_2` | Our Services 2 | " |
| `our_services_2__description` | Our Services 2 - Description | " |
| `our_services_3` | Our Services 3 | " |
| `our_services_3__description` | Our Services 3 - Description | " |
| `our_services_4` | Our Services 4 | " |
| `our_services_4__description` | Our Services 4 - Description | " |

## Components

### `lib/a2p_values.py` (new)
- `A2P_VALUE_KEYS`: ordered list of `(slug, canonical_name, category)` for the 18
  values above (`category ∈ {"fact", "copy"}`). Single source of truth.
- `make_client(location_id, refresh_token=None) -> GHLClient`: wraps
  `engine.TokenManager` (+ `set_refresh_token` when a non-DLF agency token is
  supplied — see the Prestige lesson).
- `fetch_custom_values(client, location_id) -> dict[slug -> {id, name, value}]`:
  `GET /locations/{loc}/customValues/`, keyed by parsed `fieldKey` slug.
- `apply_values(client, location_id, value_map: dict[slug -> str], *, dry_run=False) -> Report`:
  for each of the 18 slugs —
  - value blank/None → **skip** (record `"skipped: no value"`; never blanks an existing value)
  - slug exists on location → `PUT /locations/{loc}/customValues/{id}` body `{"name": existing_name, "value": value}` → record `"updated"`
  - slug missing → `POST /locations/{loc}/customValues/` body `{"name": canonical_name, "value": value}` → record `"created"`
  - swallow empty-200 / `JSONDecodeError` on POST (per existing engine lesson)
  - `dry_run=True` → make no calls, just classify what *would* happen
  - returns `Report` (lists: created / updated / skipped / errors; plus the resolved value_map for display)
- No copywriting and no scraping live here — pure GHL mechanics, fully unit-testable.

### `scripts/a2p_scrape.py` (new)
- `python scripts/a2p_scrape.py <url>` → prints JSON:
  `{title, meta_description, candidate_phone, candidate_email, candidate_colors[], headings[], readable_text}` (readable text truncated to a sane budget).
- Fetch with a normal browser UA; on failure print `{"error": ...}` and exit non-zero.
- Purpose: token-lean distillation so the skill feeds Claude structured JSON, not raw HTML.

### `scripts/a2p_apply.py` (new)
- Thin CLI: `python scripts/a2p_apply.py <location_id> <value_map.json> [--dry-run] [--refresh-token=TOKEN]`
- Loads the value-map JSON (`{slug: value, ...}`), calls `lib.a2p_values.apply_values`, prints the `Report` (human-readable + a machine block).
- Exit non-zero if any errors.

### `.claude/commands/a2p-fill-values.md` (new skill)
Args: `<business_url> <location_id> [client_slug_or_brief_path] [--refresh-token=…]`
Flow:
1. Run `scripts/a2p_scrape.py <url>` → distilled business JSON.
2. If a client slug / brief path is given, read `clients/<slug>/overview.md` and `clients/<slug>/services.md` (whichever exist).
3. Claude composes the value map: resolve the 7 facts by the priority rule; write the 11 copy values from the scraped JSON + brief in DLF voice. Write to `/tmp/a2p-values-<location_id>.json`.
4. `python scripts/a2p_apply.py <loc> /tmp/a2p-values-<loc>.json --dry-run [--refresh-token=…]` → show the user the full proposed value map (facts + copy) and the dry-run report; explicitly call out any flagged/blank facts.
5. On user approval → re-run `a2p_apply.py` without `--dry-run`.
6. Log the completion to `drleadflow/blade-ops/contents/logs/dev-activity.json` per the repo `CLAUDE.md`.

## Data flow

```
url ─▶ a2p_scrape.py ─▶ business.json ─┐
client_slug ─▶ clients/<slug>/*.md ────┼─▶ Claude builds value_map.json ─▶ a2p_apply.py ─▶ lib.a2p_values.apply_values ─▶ GHL API (POST/PUT per value) ─▶ Report
                                       ┘                                      (dry-run first, then for real)
```

## Error handling

- Blank/unresolved value → skip + surface in report; never overwrite a good value with empty.
- `business_profile__legal_business_name` not derivable from brief → blank + prominent flag.
- GHL `401` → surface clearly; if it's a non-DLF agency, instruct to pass `--refresh-token` (engine handles refresh; this is about which refresh token wins).
- GHL empty-`200` on POST → swallowed (`JSONDecodeError` handling already in engine; mirror it here if calling raw).
- Scrape failure (site down / JS-only) → continue with brief-only; flag which facts couldn't be sourced.
- Re-runs are idempotent (PUTs the same values; creates nothing new).

## Testing

- `tests/test_a2p_values.py` — `apply_values` against a fake `GHLClient` that records calls: asserts `POST` for missing slugs, `PUT` (with existing name) for present ones, skip for blanks, correct `Report` shape, empty-200 tolerated. Style mirrors `tests/verify_*.py`.
- `scripts/a2p_scrape.py` — fixture test: a saved HTML file in → asserts extracted `title` / `candidate_email` / a `candidate_color`.
- Manual smoke: `python scripts/a2p_apply.py 9uog1fOUDtxkSr4ZPx1i <prestige_current_values>.json --dry-run` → every one of the 18 classifies as `"updated"` (or `"skipped: no value"` for the genuinely-empty ones like `primary_color`), proving fieldKey matching works against real data.

## Open assumptions (verify during planning, don't block the build)

1. GHL renders `{{ custom_values.* }}` inside opt-in form checkbox labels (if not, the form fields are a separate follow-up — not this skill's job).
2. `POST /locations/{loc}/customValues/` with `{name, value}` is the create endpoint and `PUT /locations/{loc}/customValues/{id}` is the update endpoint (observed in the browser session; confirm shape when wiring the lib).
3. The location's purchased phone number is retrievable via an API the engine token can reach (fallback for `business_profile__phone_number`); if not, that fact just gets flagged like the legal name.
